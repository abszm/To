# 音频转码 Docker 应用 - 需求文档

**文档版本**: 1.0  
**创建日期**: 2026-04-27  
**状态**: 草稿

## 1. 项目概述

构建一个轻量级、容器化的自动化音频转码服务。该应用旨在通过监听指定的输入目录，自动将高保真音频（如 FLAC/WAV）转码为移动端兼容的高效格式（如 AAC/ALAC），并支持自定义输出质量和元数据保留。

## 2. 核心功能需求

### 2.1 格式自动转换
- **输入格式**: flac, wav, ogg, opus 等无损/有损音频格式
- **输出格式**: m4a (AAC) 或 alac (Apple Lossless)
- **转换质量**: 支持自定义比特率配置

### 2.2 监控与自动化
- 使用 inotify 或轮询机制实时监控输入文件夹
- 新文件进入时自动触发转码队列
- 支持批量文件处理

### 2.3 元数据完整性
- 保留原文件的 ID3/Vorbis Tags
- 保留元数据字段：艺人、专辑、封面图、曲目编号等
- 确保转码后元数据无损迁移

### 2.4 多线程并发控制
- 允许设置最大并行任务数
- 防止 Docker 容器占用过多 CPU 资源
- 避免影响宿主机其他服务运行

### 2.5 静默处理
- 容器运行过程中不进行任何交互
- 所有操作日志输出至标准输出 (stdout)
- 便于配合 `docker logs` 进行监控

## 3. 技术选型需求

### 3.1 基础镜像
- 推荐 debian:bookworm-slim 或 alpine:latest
- 目标：保持镜像体积最小化

### 3.2 核心引擎
- 使用 FFmpeg 作为转码底座
- 可选：编译支持 libfdk_aac 的 FFmpeg（高性能场景）

### 3.3 开发语言
- **选项 A**: Python (配合 watchdog 库) - 更易于处理复杂的元数据迁移
- **选项 B**: Shell Script (配合 inotify-tools)

### 3.4 容器化配置

#### 3.4.1 Volume 挂载
- 输入目录通过 `-v` 挂载至宿主机
- 输出目录通过 `-v` 挂载至宿主机

#### 3.4.2 环境变量
- `CONVERT_FORMAT`: 输出格式（例如 aac 或 alac）
- `BITRATE`: 比特率（例如 320k）
- `MAX_THREADS`: 最大并发数（例如 2）
- `DELETE_SOURCE`: 可选，是否在成功后删除原文件

## 4. 架构逻辑流程

1. **监听阶段**: 容器启动后，监听 /input 目录
2. **触发阶段**: 检测到新文件，判断后缀名
3. **处理阶段**: 调用 FFmpeg 执行转码
4. **校验阶段**: 验证输出文件是否存在
5. **清理阶段**: 若成功则触发清理或归档任务

## 5. 性能与资源控制

### 5.1 CPU 限制
- 在 docker-compose.yml 中使用 `cpus: "1.0"` 限制容器算力

### 5.2 优先级调整
- 利用 nice 命令在容器内运行转码进程
- 确保转码优先级低于系统核心服务

### 5.3 文件系统优化
- 建议将 input 和 output 目录放置在同一磁盘分区
- 利用 mv 操作的原子性
- 避免跨磁盘复制带来的 I/O 损耗

## 6. 验收标准

- [ ] 新音频文件放入 input 目录后，自动触发转码
- [ ] 转码后的文件输出到 output 目录
- [ ] 元数据（艺人、专辑、封面等）完整保留
- [ ] 支持并发控制，不占用过多 CPU
- [ ] 支持通过环境变量配置输出格式、比特率等
- [ ] 所有日志可透过 `docker logs` 查看
- [ ] 可选：转码成功后自动删除源文件

## 7. 约束条件

- 容器必须轻量级，镜像体积尽可能小
- 支持长时间稳定运行
- 不依赖外部服务，独立运行
- 日志输出清晰，便于故障排查
